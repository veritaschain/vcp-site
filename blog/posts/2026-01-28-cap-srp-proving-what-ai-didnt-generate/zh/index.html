CAP-SRP：证明AI未生成的内容——AI内容真实性的缺失层

为什么现有框架无法回答最关键的问题，以及IETF SCITT如何实现可验证的拒绝证明

2026年1月28日 40分钟阅读 技术分析
C2PA EU AI Act SCITT
执行摘要

2026年1月的xAI/Grok危机暴露了AI内容真实性的关键空白：现有标准只能证明AI创建了什么，却无法证明AI拒绝创建什么。 本文分析了为什么这一空白的存在，介绍了CAP-SRP（内容真实性协议-安全拒绝来源）作为解决方案， 以及组织如何在2026年8月EU AI Act合规截止日期之前准备实施。

2026年1月：一次危机揭示了缺失的架构层

xAI/Grok图像危机（2026年1月）

当用户在Grok聊天机器人上传播"绕过安全防护技术"时，一场安全危机迅速升级。 在三天的时间跨度内，曾被安全护栏阻止的图像生成请求开始产生有害输出。 更糟的是——xAI无法提供任何可独立验证的记录来证明安全系统曾经正常运行。

这一事件不仅仅是个别失误，它揭示了整个AI治理堆栈中的架构空白：

"唯一比安全系统失败更糟糕的，是无法证明它曾经工作过。"

为什么C2PA和现有标准无法解决这个问题

以C2PA（内容来源和真实性联盟）为代表的现有标准为已创建的内容建立了良好的出处证明机制。 然而，它们在设计上只关注"正向路径"——当内容被成功生成时附加元数据。

能力 C2PA CAP-SRP
证明AI生成了特定内容 ✅ 支持 ✅ 支持
证明AI拒绝生成特定请求 ❌ 不支持 ✅ 支持
验证安全系统正常运行的证明 ❌ 不支持 ✅ 支持
检测安全绕过尝试的时间窗口 ❌ 不支持 ✅ 支持
提供独立可验证的拒绝证明 ❌ 不支持 ✅ 支持
关键洞察：完整性缺口

当监管机构询问"你的安全系统在过去24小时内是否正常运行？"时，仅有C2PA风格的证明意味着：
• 您只能展示已生成的内容
• 您无法证明被拒绝请求的存在
• 您无法证明安全系统正在主动工作

CAP-SRP：安全拒绝来源协议

CAP-SRP（内容真实性协议-安全拒绝来源）是专门为填补这一空白而设计的扩展协议。 它建立在IETF SCITT（供应链完整性、透明度和信任）架构之上，为AI系统的安全拒绝行为创建密码学可验证的证明。

核心设计原则

原则1：完整性保证

每个生成尝试必须有记录的结果——生成、拒绝或错误。 没有请求可以"消失"而不留下密码学证据。 

enum GenerationOutcome {
    GENERATED,     // 内容已创建
    DENIED,        // 安全系统拒绝请求
    ERROR,         // 系统错误阻止处理
    TIMEOUT        // 请求超时未完成
}

原则2：时序锚定

关键创新是事件在安全评估之前被记录。这意味着即使安全系统被绕过， 原始请求的存在证明仍然存在于透明度日志中。

T₀：请求接收

请求被记录到SCITT透明度日志（不可变）

T₁：安全评估

安全系统评估请求并做出决定

T₂：结果记录

决定（生成/拒绝/错误）被记录，与T₀事件关联

T₃：外部锚定

批次根哈希通过TSA或区块链进行外部验证

原则3：隐私保护

CAP-SRP不要求存储原始提示词内容。相反，它使用：

  • 提示词哈希：SHA-256哈希而非原始文本
  • 风险类别：安全拒绝的分类（如CSAM、暴力、仇恨言论）而非具体内容
  • 密码学碎片化：支持GDPR删除权的密钥销毁机制
  • 可选zk-STARK集成：允许证明拒绝存在而不暴露任何请求细节

技术架构

CAP-SRP利用IETF SCITT架构提供以下核心能力：

1. 追加式透明度日志

基于RFC 6962 Merkle树实现的透明度日志确保：

  • 事件一旦记录就无法删除或修改
  • 任何第三方都可以验证日志的一致性
  • 包含证明允许高效验证特定事件的存在
// SCITT语句结构（简化）
{
    "statement_type": "refusal_event",
    "issuer": "xai.com/grok",
    "timestamp": "2026-01-15T14:30:00Z",
    "subject": {
        "request_hash": "sha256:a1b2c3d4...",
        "outcome": "DENIED",
        "risk_category": "VIOLENCE",
        "safety_model_version": "grok-safety-v2.3"
    },
    "signature": "ed25519:..."
}

2. 数字签名

使用Ed25519签名（具有向后量子迁移路径）确保：

  • 事件确实来自声称的AI系统
  • 事件内容在签名后未被篡改
  • 签名可以被任何第三方独立验证

3. 外部锚定

批次根哈希通过以下方式进行外部验证：

  • RFC 3161时间戳机构（TSA）：提供法律认可的时间证明
  • OpenTimestamps：利用比特币区块链的分布式时间锚定
  • 多重锚定：同时使用多种方式增强可靠性

监管背景：为什么2026年8月是关键截止日期

监管时间线

EU AI Act第12条要求高风险AI系统保持"自动生成的日志"， 这些日志必须"允许追溯与其预期目的相关的系统运行"。 高风险条款于2026年8月2日生效。

法规如何要求可验证的拒绝证明

法规 条款 要求 CAP-SRP映射
EU AI Act 第12条 自动日志记录与追溯能力 SCITT透明度日志
EU AI Act 第72-73条 上市后监控与事件报告 拒绝事件追踪
加州AI透明法 SB 942 生成内容披露 生成/拒绝完整记录
ESMA AI指南 ESMA35-335435667-5924 投资服务中AI使用的文档记录 事件完整性保证

"证明安全系统工作"的问题

考虑以下监管场景：

"请提供证据证明贵公司的AI安全系统在2026年1月1日至15日期间正常运行， 并且所有被归类为有害的生成请求都被正确拒绝。"

没有CAP-SRP：组织只能提供内部日志，这些日志可能被篡改、删除或简单地不存在。 无法提供独立可验证的证据证明安全系统确实在工作。

有CAP-SRP：组织可以提供：

  • SCITT透明度日志中所有拒绝事件的密码学证明
  • 外部时间戳证明事件发生在声称的时间
  • 一致性证明显示日志未被篡改
  • 包含证明允许监管机构验证特定事件

实施路线图

阶段1：评估（4-6周）

  • 审计现有AI生成系统的日志能力
  • 识别所有生成端点和安全检查点
  • 评估当前拒绝事件的可追溯性
  • 映射数据保留和隐私要求

阶段2：基础设施（6-8周）

  • 部署SCITT兼容的透明度日志基础设施
  • 集成Ed25519签名到生成管道
  • 建立与TSA/OpenTimestamps的外部锚定
  • 实施隐私保护的哈希和分类系统

阶段3：集成（4-6周）

  • 修改AI生成管道以记录所有结果
  • 实施"先记录后评估"的时序流程
  • 建立监控和警报系统
  • 进行端到端验证测试

阶段4：验证（持续）

  • 定期审计日志完整性
  • 独立第三方验证
  • 监管报告能力测试
  • 持续改进和更新

从"信任我们的安全系统"到"验证我们的安全系统"

xAI/Grok危机是一个警钟，但也是一个机会。它清楚地表明，AI治理需要从基于信任的模型 转向基于验证的模型。CAP-SRP提供了实现这一转变的技术基础。

核心价值主张

对于AI提供商：可证明的安全合规，减少监管风险，建立用户信任

对于监管机构：独立可验证的证据，高效的审计流程，可追溯的事件链

对于用户：透明的安全运营，可验证的保护承诺，问责制保障

立即行动

距离EU AI Act高风险条款生效还有不到7个月。组织现在需要：

  1. 评估当前状态：了解现有拒绝事件的可追溯性
  2. 规划基础设施：为SCITT部署做准备
  3. 试点实施：在低风险环境中测试CAP-SRP
  4. 参与标准制定：加入IETF SCITT工作组
