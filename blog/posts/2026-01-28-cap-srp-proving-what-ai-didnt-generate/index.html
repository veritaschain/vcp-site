Executive Summary

The xAI/Grok Crisis (January 2026)

Over 6,700 non-consensual sexual images per hour were generated before safeguards were circumvented. 12+ jurisdictions launched investigations. 35 U.S. state attorneys general demanded proof of safety measures.

The critical gap: xAI's refusals left no cryptographic trace. Claims of "safety filters" became legally unverifiable assertions.

While existing standards like C2PA excellently prove what AI created, nothing in the current landscape proves what AI refused to create.

CAP-SRP Solution

Content Authenticity Protocol - Safe Refusal Provenance addresses this fundamental gap by treating refusals as first-class cryptographically provable events.

The Accountability Gap

The Positive-Attestation Problem

Current content authenticity standards share a fundamental architectural limitation: they only support positive attestations—cryptographic proof that something exists or happened.

C2PA Architecture:
  ┌────────────────────────────────────────────────┐
  │              C2PA Manifest                      │
  │  ┌─────────────────────────────────────────┐   │
  │  │ Claim Generator: Adobe Firefly           │   │
  │  │ Claim Signature: COSE_Sign1 + X.509      │   │
  │  │ Assertions: Actions, ingredients, meta   │   │
  │  │ Hard Binding: SHA-256 hash of asset      │   │
  │  └─────────────────────────────────────────┘   │
  │                      │                          │
  │                      ▼                          │
  │  ┌─────────────────────────────────────────┐   │
  │  │     JUMBF Container (in JPEG/PNG/MP4)    │   │
  │  └─────────────────────────────────────────┘   │
  └────────────────────────────────────────────────┘

What C2PA CAN prove:   "This asset was created by [Entity] at [Time]" ✓
What C2PA CANNOT prove: "Request [X] was denied at [Time] for [Reason]" ✗

There is no "null manifest" in C2PA. There is no "refusal receipt." The specification contains no provision for events where no asset was created.

The Regulatory Verification Problem

The EU AI Act Article 12 requires high-risk AI systems to enable "automatic recording of events (logs) over the lifetime of the system." When regulators ask "did your system refuse to generate harmful content?", the AI provider currently has three possible responses:

Response Status Problem
"We have internal logs" ❌ Unverifiable Could be fabricated, altered, or selectively retained
"Our filters are effective" ⚠️ Trust-based A claim without evidence
"Here's our cryptographic proof" ✓ Verifiable Currently impossible — CAP-SRP enables this

Technical Limitations of Watermarking

DeMark Attack (January 22, 2026): MIT/Stanford researchers published "DeMark: A Query-Free Black-Box Attack on Deepfake Watermarking Defenses" (arXiv:2601.16473). Detection accuracy dropped from 100% to 32.9% across eight state-of-the-art watermarking schemes.

The Fundamental Problem

Watermarking operates at the content layer. A missing watermark could mean:

  • Human-created content
  • AI refused to create
  • Watermark stripped after creation
  • Non-compliant AI system

CAP-SRP Alternative: Instead of marking content (content layer), CAP-SRP records system decisions (system layer). This captures the complete decision space, including refusals.

IETF SCITT: The Foundation

The IETF Supply Chain Integrity, Transparency and Trust (SCITT) Working Group provides the foundation. The core architecture draft (draft-ietf-scitt-architecture-22) reached the RFC Editor Queue in late 2025.

Security Properties:
  ✓ Append-Only:      Statements cannot be modified, deleted, or reordered
  ✓ Non-Equivocation: No forks—everyone sees the same ordered collection
  ✓ Replayability:    Any authorized actor can verify each registration

The Completeness Invariant

The core innovation of CAP-SRP:

∀ time_window T:
  COUNT(GEN_ATTEMPT) = COUNT(GEN) + COUNT(GEN_DENY) + COUNT(GEN_ERROR)

In plain terms: Every generation attempt MUST have a recorded outcome.

This invariant enables two critical detections:

  • Hidden Results: If unmatched_attempts > 0, the system is hiding generation results
  • Fabricated Refusals: If orphan_outcomes > 0, the system is fabricating refusal records

Cryptographic Mechanisms

Merkle Tree Architecture (RFC 9162)

Metric Value
Inclusion Proof Size O(log n)
Proof for 80M entries ~3 KB
Full hash chain alternative ~800 MB

Signature Scheme: Ed25519

  • Deterministic signatures: Prevents key leakage from poor RNG
  • Complete addition law: Eliminates special-case vulnerabilities
  • Side-channel resistance: Designed against timing attacks
  • Wide adoption: Signal, OpenSSH (default since 5.7+), blockchain systems

Post-Quantum Migration Path

NIST FIPS 204 (August 2024) standardized ML-DSA (formerly Dilithium). CAP-SRP supports hybrid signatures for maximum assurance:

signature = {
  "ed25519": sign_ed25519(message),
  "ml_dsa_65": sign_ml_dsa_65(message),
  "binding": hash(ed25519_sig || ml_dsa_sig)
}

Privacy-Preserving Verification

The Prompt Privacy Problem

Refusal logging creates a tension: to verify a refusal was legitimate, you might need to see what was refused. But prompts may contain personal information, proprietary business data, or evidence of criminal intent.

CAP-SRP Principle: Log decisions, not content.

What IS logged:

  • prompt_hash = SHA-256(prompt) — Verifiable
  • risk_category = "NCII_RISK" — Categorical
  • risk_score = 0.94 — Confidence
  • policy_version = "2026-01-15" — Policy reference

What is NOT logged:

  • Raw prompt text — Privacy violation
  • User identity (beyond hash) — Privacy violation
  • Detailed content analysis — May reveal prompt content

Optional zk-STARK Integration

For scenarios requiring stronger privacy guarantees, CAP-SRP defines optional zk-STARK integration that can prove a refusal was legitimate without revealing the actual prompt.
