第一部分

当前状态的诚实陈述

VeritasChain Protocol (VCP) v1.0 的设计充分考虑了 IETF RFC 兼容性，但目前我们尚未提交 Internet-Draft，也未在任何特定的 IETF 工作组启动标准化流程。

本页面介绍 VCP 所基于的技术前提，以及我们所设想的未来标准化路径。

为什么这很重要

对标准化状态保持透明与 VCP 的核心理念一致：加密学上的真实，而非营销宣传。我们相信通过技术价值和运营记录来赢得信誉，而不是过早的公告。

第二部分

与现有RFC的技术对齐

VCP v1.0 的设计与以下 IETF RFC 兼容，确保未来可以转换为 Internet-Draft，并与其他基于 RFC 的系统实现互操作。

RFC 8785

JSON规范化方案 (JCS)

确定性JSON序列化，确保所有实现中用于加密哈希的字节序列相同。

RFC 6962

证书透明度

遵循CT模式的Merkle Tree审计设计，实现高效的包含证明和篡改检测。

draft-ietf-uuidrev

UUIDv7

时间排序、可排序、全局唯一的标识符，带有嵌入时间戳，用于事件ID。

RFC 8032

Ed25519 签名

爱德华曲线数字签名，提供高安全性、优秀性能和紧凑签名。计划未来支持PQC。

RFC 3339

时间戳

ISO 8601兼容的日期/时间格式，确保系统间一致的时间戳表示。

FIPS 180-4

SHA-256

NIST批准的安全哈希算法，用于所有哈希链和Merkle Tree操作。

这种对齐确保了未来Internet-Draft转换和与其他RFC系统的互操作性。

第三部分

迈向IETF RFC的路线图

IETF RFC标准化被定位为中长期选择，而非短期目标。我们将首先通过实际部署使VCP v1.x成熟，然后再考虑下一步。

1

实施和运营记录积累

当前阶段

积累实际部署经验，收集来自真实实现的反馈。

  • Early Adopter计划反馈收集
  • VC-Certified Silver / Gold运营经验
  • VCP Explorer / API生产部署
2

社区草案发布

计划中

基于vcp-spec，在GitHub上发布遵循IETF Internet-Draft模板的v0.x草案，供社区审查。

预期格式: draft-vso-vcp-00.txt
3

IETF社区参与

计划中

与相关IETF工作组对话，并可能参与BoF (Birds of a Feather) 会议。

潜在工作组

OPSAWG, SEC Area, RTG Area, ART Area

备选路径

如无合适WG，可通过BoF会议 → 组建新WG

4

考虑提交Internet-Draft

未来

在获得足够的采用数据和社区反馈后，考虑向 datatracker.ietf.org 正式提交Internet-Draft。

5

WG采纳与RFC流程

目标

如被工作组采纳：迭代修订、继续保持运行代码证据、IETF Last Call → IESG审核 → 最终RFC发布。

IETF按照"大致共识和运行代码"原则运作 — 部署记录至关重要。
第四部分

塑造未来标准

VCP的标准化过程高度依赖于实施者和用户的反馈。

参与Early Adopter计划或获得VC-Certified认证的企业将对未来Internet-Draft和RFC的内容具有实质性影响力。

这是一个不仅仅"遵循"标准，而是"参与设计"的机会 — 这是VCP生态系统的关键差异化特征。

实施者

您的实施反馈直接影响规范改进

VC认证企业

认证采用者在标准化讨论中获得发言权

监管机构

监管输入确保从第一天起就符合合规要求

潜在 IANA 注册表

如果 VCP 被标准化，可能申请以下注册表：

VCP Event Type Registry

OrderSubmitted, OrderFilled, OrderCanceled 等

VCP Hash Algorithm Registry

SHA-256, SHA-384, SHA-512, SHA3-256 等

VCP Signature Algorithm Registry

Ed25519, ECDSA-P256, RSA-2048, Dilithium2, Falcon-512

注册表详情将在 I-D 流程中确定。
